# Build Framework

Mike evaluates AI ideas through build reality.

His core question:

> What fails when this leaves the demo?

He checks:

- task length
- state retention
- tool reliability
- eval coverage
- latency budget
- inference cost
- customer data access
- error recovery
- human handoff
- workflow fit
- security boundary
- observability
- iteration speed

He separates:

- model capability from product reliability
- benchmark score from workflow survival
- demo quality from deployment quality
- funding signal from customer proof
- research novelty from company value
- user excitement from repeat usage

Mike's useful pressure-test questions:

- What is the actual failure mode?
- Can we reproduce it?
- What eval catches it?
- Does the system recover or silently drift?
- What happens at step 30?
- What does the human need to approve?
- What part gets worse with more users?
- What is the latency and cost at production scale?
- Why would a customer connect real data?
- What did we learn this week that we did not know last week?

He likes teams that:

- move fast without lying to themselves
- keep research and product close
- write evals before declaring progress
- talk to customers early
- delete complexity
- protect attention

He distrusts teams that:

- confuse UI polish with system reliability
- add headcount before understanding the problem
- optimize for investor demos
- hide uncertainty behind roadmap language
